## Security Issues

### 1. `PUT /auth/change-password` — no `auth` middleware (High)
The route accepts `email` in the body to identify the user instead of reading from `req.user`. It works because it still checks the old password, but it is inconsistent with every other protected route and opens the door to enumeration if error messages ever diverge. Should be protected by `auth` and use `req.user` to identify the caller.

### 2. `POST /user/seed-super-admin` — completely unprotected (High)
No auth, no rate limiting, no environment guard. Any unauthenticated request can attempt to seed an admin. The service blocks it if an admin already exists, but a race condition or a fresh deploy without the seed still exposes this. Should be disabled in production (env check) or removed after first use.

### 3. `findUserWithEmailService` — does not filter deleted/inactive users (Medium)
Used in the `auth` middleware and login flow. Deleted (`isDeleted: true`) or deactivated (`isActive: false`) users can still be found, their tokens verified, and they can proceed past the auth middleware (which does check `isActive`). However, a soft-deleted user whose `isActive` was not also set to `false` would pass auth entirely. The service should always filter `{ isDeleted: false }`.

### 4. `POST /auth/send-verification-email` — redundant email in body (Low)
The route is already protected by `auth`, which attaches `req.user.email`. The controller then reads a second `email` from `req.body` and verifies they match in the service. The body email is unnecessary and expands the attack surface. Use `req.user.email` directly.

## Validation Issues

### 1. `changePasswordSchema` — `newPassword` has no length constraints (Medium)
`resetPasswordSchema` enforces `min(8)` and `max(20)` on `newPassword`, but `changePasswordSchema` has none. A user could set an empty string as their new password through `PUT /auth/change-password`.

```ts
// current
newPassword: z.string({ message: "New password is required!" }),

// should be
newPassword: z
  .string({ message: "New password is required!" })
  .min(8, { message: "Password must be at least 8 characters!" })
  .max(20, { message: "Password must be at most 20 characters!" }),
```

### 2. No password complexity validation anywhere (Low)
Neither login, change-password, nor reset-password enforce any complexity rules (uppercase, number, symbol). Minor for an internal tool, but worth noting.

## Consistency Issues

### 1. `PUT` vs `PATCH` for mutation routes (Low)
`PUT /auth/change-password` uses `PUT` but all other mutation routes use `PATCH`. `PATCH` is more semantically correct for partial updates. Consider changing to `PATCH /auth/change-password`.

### 2. `seed-super-admin` uses the same schema as `create-user` (Low)
`createUserSchema` is shared for both routes, but `seed-super-admin` also requires a `password` field in the body, which `createUserSchema` does not validate. The controller destructures `password` from `req.body` but validation does not check it — an empty or missing password silently passes Zod and gets hashed as-is by bcrypt.
